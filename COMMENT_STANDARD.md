# Flutter/Dart 分级注释规范指南

本规范旨在通过 **“分级策略”** 平衡开发效率与代码可维护性。我们拒绝“一刀切”的繁琐，主张根据代码的重要性与复杂度，选择合适的注释深度。

## 0. 执行约定（与仓库规范对齐）

- 本文件是注释规范唯一来源，`AGENTS.md` 中涉及注释的条款以本文件为准。
- 注释与提示统一使用中文，文件编码统一为 UTF-8。
- 快交付场景下，默认采用“能不写就不写、必须写就写清楚”的策略：
  - 核心链路与公共能力必须注释（方案 A）。
  - 普通 UI 与简单逻辑保持简洁（方案 B）。
  - 生成文件（`*.g.dart`、`*.freezed.dart`、`lib/generated/`）不添加人工注释。

---

## 1. 核心策略：分级注释

我们将注释分为两个等级：**方案 A（详细注释）** 和 **方案 B（简单注释）**。

| 等级 | 适用场景 | 特点 |
| :--- | :--- | :--- |
| **方案 A: 详细模式** | **核心业务、公共组件、工具类、API封装** | 结构化、标签化、包含完整逻辑链，供 AI 和团队深度维护。 |
| **方案 B: 简单模式** | **普通 UI 页面、简单控制器、数据模型** | 自然语言、简洁明了、一句话概括，注重阅读流畅性。 |

---

## 2. 方案 A：详细注释 (Detailed Mode)

**适用对象**:
- 被广泛引用的 `Utils` 类
- 复杂的 `Service` / `Controller` 核心逻辑
- 公共 UI 组件 (`CommonWidget`)
- 涉及金钱、权限、硬件交互的关键代码

### 2.1 详细模式模板
必须包含 **意图**、**参数** 和 **逻辑步骤**。

```dart
/// [功能]: 执行登录并处理复杂的 Token 刷新逻辑。
/// [输入]: 
///   - [username]: 用户账号
///   - [forceRefresh]: 是否强制刷新 Token (默认 false)
/// [返回]: 成功返回用户信息，失败抛出 [AuthException]。
/// 
/// [业务逻辑]:
///   1. 检查本地 Token 是否过期。
///   2. 如果过期且允许刷新，调用 Refresh Token 接口。
///   3. 失败重试 3 次 (指数退避策略)。
///   4. 最终失败则清空状态并跳转登录页。
Future<User> secureLogin(String username, {bool forceRefresh = false}) async { ... }
```

### 2.2 复杂数据结构
对于 `Map` 或 `dynamic` 类型，必须写出 JSON 结构。

```dart
/// [数据结构]:
/// ```json
/// {
///   "id": 1001,
///   "roles": ["admin", "editor"]
/// }
/// ```
```

## 3. 方案 B：简单注释 (Lite Mode)

**适用对象**:
- 普通的业务页面 (`View`)
- 简单的交互逻辑 (`onTap`)
- 仅在当前模块使用的辅助函数
- 数据实体类 (`Entity`)

### 3.1 简单模式模板
使用自然语言，一句话说清楚**做什么**。不需要 `[功能]` 这种标签。

**UI 构建方法**:
```dart
/// 构建用户名输入框
Widget _buildUserNameInput() { ... }
```

**简单交互**:
```dart
/// 点击登录，校验通过后跳转首页
void onLoginPressed() { ... }
```

**数据模型**:
```dart
/// 用户收货地址实体
class AddressEntity { 
  /// 详细地址 (包含省市区)
  final String fullAddress; 
  /// 是否为默认地址
  final bool isDefault;
}
```

## 4. 通用规则 (适用于所有场景)

无论使用哪种模式，以下规则必须遵守：

### 4.1 关键约束 (Constraint)
对于**不可修改**或**有特殊背景**的代码，必须加标记。

```dart
// [CONSTRAINT]: iOS 审核要求必须先弹窗同意协议，否则会被拒审。
if (Platform.isIOS && !isAgreed) return;
```

### 4.2 TODO & FIXME
```dart
// TODO(张三): 下个版本需要支持多语言切换
```

### 4.3 什么时候不写注释？
- 显而易见的 Get/Set 方法。
- 标准的 Flutter 生命周期 (`initState`, `build`, `dispose`)，除非有特殊逻辑。
- 只有一行代码的简单闭包。

---

## 5. 总结

- **写公共类时**：想象你正在写开源库的文档（用方案 A）。
- **写业务页面时**：想象你正在给同事留便条（用方案 B）。
- **遇到坑时**：大声告诉全世界（用 `[CONSTRAINT]`）。
